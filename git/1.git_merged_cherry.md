# 🎯 Scenario Setup

## Branch History

```text
A --- B --- C ----------- M   (develop)
       \                 /
        D ----- E -------
```

👉 Meaning:

* Feature branch created from **B**
* Feature commits: **D, E**
* Develop moved ahead: **C**
* Then merged → **M**

---

# 🎯 Now You Run

```bash
git cherry-pick -m 1 M
```

---

# 🧠 Step-by-Step What Git Does Internally

## 1️⃣ Identify Parents of Merge Commit

Merge commit **M has 2 parents**:

```text
Parent 1 = C (develop side)
Parent 2 = E (feature side)
```

---

## 2️⃣ Apply `-m 1`

```bash
-m 1
```

👉 Means:

> “Use Parent 1 (C) as base”

---

## 3️⃣ Git Calculates Diff

Git computes:

```text
M - C
```

👉 What is `M - C`?

```text
= changes introduced by feature branch
= (D + E)
```

---

# 🎯 Final Result in Your Release Branch

After cherry-pick:

```text
release branch gets:

✔ Changes from D
✔ Changes from E
❌ NOT commit C
❌ NOT merge structure
```

---

# 🔥 Real Visualization

## Before cherry-pick:

```text
release:
A --- B
```

---

## After:

```text
release:
A --- B --- (D + E applied as ONE commit)
```

👉 Important:

* It becomes **one new commit**
* Not D and E separately
* Not M exactly

---

# ⚡ What Actually Gets Created?

Git creates a **new commit**:

```text
R1 (new commit in release)
```

Where:

```text
R1 = diff (M - C)
```

---

# 🧠 Mental Model (Super Important)

Think:

```text
M = Final Result (C + D + E)
C = Base
```

👉 So:

```text
git cherry-pick -m 1 M
= Remove base (C)
= Apply only new changes (D + E)
```

---

# 🔥 Example with Code (Simple)

### Feature branch:

```js
// D
function add() { return 1 + 1; }

// E
function sub() { return 2 - 1; }
```

### Develop added:

```js
// C
function mul() { return 2 * 2; }
```

---

### Merge result (M):

```js
function mul() { return 2 * 2; }
function add() { return 1 + 1; }
function sub() { return 2 - 1; }
```

---

## Now cherry-pick:

```bash
git cherry-pick -m 1 M
```

👉 Result in release:

```js
function add() { return 1 + 1; }
function sub() { return 2 - 1; }
```

❌ `mul()` NOT included (because it came from C)

---

# ⚠️ Important Edge Case

If your release branch:

```text
A --- B
```

👉 Then applying D+E may:

* conflict
* break (if D/E depend on C)

---

# 🚀 Final Summary

| Concept            | Result                   |
| ------------------ | ------------------------ |
| `-m 1`             | Use develop side as base |
| Applied changes    | Only feature (D + E)     |
| Commit C           | ❌ Not included           |
| New commit created | ✅ Yes                    |
| Safe?              | Only if base compatible  |

---

# ⚡ Golden Rule

👉 **`git cherry-pick -m 1 M` = “Give me feature changes only, ignore base branch changes”**

Great — now you’re asking the **most important part** of merge cherry-pick 🔥
Let’s make `-m 2` crystal clear.

---

# 🎯 Same Scenario

```text
A --- B --- C ----------- M   (develop)
       \                 /
        D ----- E -------
```

👉 Merge commit **M has 2 parents**:

* **Parent 1 = C (develop side)**
* **Parent 2 = E (feature side)**

---

# 🎯 Command

```bash
git cherry-pick -m 2 M
```

---

# 🧠 What `-m 2` Means

👉 It tells Git:

> “Use **Parent 2 (feature branch)** as base”

---

# 🔥 Internal Calculation

Git does:

```text
M - Parent2 (E)
```

👉 So:

```text
M - E = changes from develop side
```

---

# 🎯 Final Result

After running:

```bash
git cherry-pick -m 2 M
```

👉 Your release branch gets:

```text
✔ Changes from C (develop side)
❌ NOT D, E (feature changes)
```

---

# 🔥 Visual Understanding

## Merge commit contains:

```text
M = C + D + E
```

---

## With `-m 2`:

```text
Remove feature base (E)
Apply remaining → C
```

---

# 🧠 Mental Model (Never Forget)

Think:

```text
M = Pizza 🍕 (base + toppings)

C = Cheese
D, E = toppings
```

👉 `-m 1`:

```text
Remove cheese → keep toppings 🍕
```

👉 `-m 2`:

```text
Remove toppings → keep cheese 🧀
```

---

# 🔥 Practical Example (Code)

### Merge result (M):

```js
function mul() { return 2 * 2; }   // from C
function add() { return 1 + 1; }   // from D
function sub() { return 2 - 1; }   // from E
```

---

## Now run:

```bash
git cherry-pick -m 2 M
```

👉 Result:

```js
function mul() { return 2 * 2; }
```

❌ `add()` and `sub()` are NOT included

# ⚠️ Real-World Usage

👉 Almost **never used** ❌

Why?

* You usually want **feature changes**
* Not base branch changes

---

# 🚀 When could `-m 2` be useful?

Rare cases like:

* You want to **extract only develop-side changes**
* Undo feature impact
* Debugging merge issues

---

# ⚡ Final Comparison

| Command | Result                    |
| ------- | ------------------------- |
| `-m 1`  | ✔ Feature changes (D + E) |
| `-m 2`  | ✔ Develop changes (C)     |

# 🧠 One-Line Memory Trick

```text
-m 1 → keep feature
-m 2 → keep base branch
```

# 🔥 Final Takeaway

👉 If your goal is:

* Feature → use `-m 1` ✅
* Base branch → use `-m 2` ⚠️ (rare)
