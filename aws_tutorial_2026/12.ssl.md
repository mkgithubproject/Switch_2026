# Government-approved ID card

# HTTP vs HTTPS

---

# HTTP

```text id="40mrkj"
Browser  ←→  Server
```

Data is plain text.

Anyone can read it.

Example:

* passwords
* tokens
* API data
* credit card

can be stolen.

---

# HTTPS

```text id="1pkpt5"
Browser  ←encrypted→  Server
```

Now data is encrypted.

Attackers see garbage text.

---

# HTTPS Uses

# TLS/SSL

TLS is modern version of SSL.

People still say SSL certificate.

---

# What is Certificate?

A certificate contains:

| Field       | Example               |
| ----------- | --------------------- |
| Domain      | myapp.com             |
| Public Key  | encryption key        |
| Issued By   | Certificate Authority |
| Expiry Date | validity              |
| Signature   | CA verification       |

---

# Certificate Authority (CA)

Trusted organizations.

Examples:

* DigiCert
* Let's Encrypt
* Sectigo

Browsers already trust them.

---

# Core Concept

# Public Key + Private Key

Very important.

Server creates:

```text id="3vynol"
Public Key
Private Key
```

---

# Public Key

Can be shared with everyone.

Used for:

* encryption
* verification

---

# Private Key

Secret.

Stored only on server.

Used for:

* decryption
* signing

Never shared.

# Step-by-Step HTTPS Flow

## Step 1 — Browser Connects

```text id="yjlk94"
Browser → Server
"Hello"
```

---

## Step 2 — Server Sends Certificate

Server sends:

* certificate
* public key

```text id="yyiv9r"
Server → Browser
"Here is my certificate"
```

---

## Step 3 — Browser Verifies Certificate

Browser checks:

✅ trusted CA?
✅ expired?
✅ domain matches?
✅ signature valid?

If valid:

🔒 browser shows lock icon

---

## Step 4 — Browser Creates Session Key

Browser generates:

```text id="t46m0n"
Random symmetric key
```

called:

# Session Key

---

# Why Session Key?

Public/private encryption is slow.

So HTTPS uses:

* asymmetric encryption initially
* symmetric encryption afterwards

---

# Step 5 — Browser Encrypts Session Key

Using server public key:

```text id="pl88j5"
Encrypted(SessionKey)
```

Only server private key can decrypt it.

---

## Step 6 — Server Decrypts

Using private key.

Now both know same session key.

---

## Step 7 — Secure Communication Starts

Now all traffic encrypted using session key.

```text id="2ot7jd"
Browser ⇄ Encrypted Data ⇄ Server
```

---

# Never Forget Memory Trick

# Certificate = Identity

# Public Key = Lock

# Private Key = Secret Key

# Session Key = Fast temporary password

# End-to-End Encryption

Sometimes companies want:

```text id="3nyqj7"
Browser ⇄ HTTPS ⇄ ALB ⇄ HTTPS ⇄ EC2
```

* certificates also installed on EC2

# Difference

| Type            | Description               |
| --------------- | ------------------------- |
| SSL Termination | HTTPS only till ALB       |
| End-to-End SSL  | HTTPS till backend server |

# Most Important Interview Questions

---

# What is SSL termination?

Load balancer decrypts HTTPS traffic and forwards HTTP internally.

---

# Why use ACM?

* free certificates
* auto renewal
* easy ALB integration

---

# Where is certificate installed with ALB?

On ALB listener.

---

# Does EC2 need certificate with ALB?

Usually no.

Only if using end-to-end HTTPS.

---

# Why HTTPS needed?

* encryption
* authentication
* data integrity

---

# What happens during TLS handshake?

Browser:

* verifies certificate
* exchanges session key
* establishes encrypted communication
