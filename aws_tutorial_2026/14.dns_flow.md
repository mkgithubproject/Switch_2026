# Chapter 1: Why DNS Exists

Suppose you create a Node.js server:

```javascript
const http = require("http");

http.createServer((req, res) => {
  res.end("Hello");
}).listen(3000);
```

Your server runs on:

```text
54.23.11.100
```

Users can access it using:

```text
http://54.23.11.100
```

But humans don't like remembering:

```text
54.23.11.100
104.26.10.78
142.250.183.14
```

So we invented names:

```text
google.com
amazon.com
xyz.com
```

Now we need a system that converts:

```text
xyz.com
      ↓
54.23.11.100
```

This system is DNS.

---

# Chapter 2: The Simplest DNS Possible

Imagine there is only ONE website in the world.

```text
xyz.com → 54.23.11.100
```

We could store it in one file:

```text
xyz.com = 54.23.11.100
```

When browser asks:

```text
What is xyz.com?
```

We return:

```text
54.23.11.100
```

Easy.

---

# Problem

Now there are billions of domains.

```text
google.com
amazon.com
facebook.com
netflix.com
xyz.com
```

One file won't work.

One server won't work.

So DNS became hierarchical.

---

# Chapter 3: DNS Hierarchy

DNS is like a tree.

```text
                    .
                 (ROOT)

               /   |   \
            com   org   in

             |
          xyz.com

             |
         api.xyz.com
```

Every DNS lookup starts at:

```text
.
```

called the **Root Zone**.

---

# Chapter 4: Root Server

Root servers know only:

```text
Who manages .com?

Who manages .org?

Who manages .in?
```

Think:

```text
Root = Reception Desk
```

Example:

Resolver asks:

```text
Where is xyz.com?
```

Root says:

```text
I don't know.

But .com department knows.
```

---

# Chapter 5: TLD Server

TLD = Top Level Domain

Examples:

```text
.com
.org
.in
.net
.io
```

The .com servers know:

```text
Which nameservers manage each .com domain
```

Example database:

```text
google.com → ns1.google.com

amazon.com → ns1.amazon.com

xyz.com → ns-100.awsdns.com
```

Notice:

No IPs yet.

Only nameservers.

---

# Chapter 6: Domain Registrar

You buy:

```text
xyz.com
```

from GoDaddy.

GoDaddy talks to the `.com` registry and creates:

```text
xyz.com
```

in the .com database.

Initially:

```text
xyz.com

NS:
ns01.domaincontrol.com
ns02.domaincontrol.com
```

---

# Chapter 7: What Is a Nameserver?

A nameserver is a DNS server that knows the actual records.

Example:

```text
ns-100.awsdns.com
```

This nameserver may answer:

```text
xyz.com → 54.23.11.100

api.xyz.com → 54.23.11.101
```

Think:

```text
Nameserver = Librarian
```

---

# Chapter 8: What Is a Hosted Zone?

Inside Route53:

You create:

```text
Hosted Zone: xyz.com
```

It contains:

```text
xyz.com      A      54.23.11.100

api.xyz.com  A      54.23.11.101

mail.xyz.com MX     Google Mail
```

Think:

```text
Hosted Zone = Folder of DNS Records
```

---

# Chapter 9: Route53 Setup

AWS creates:

```text
Hosted Zone

xyz.com
```

and gives:

```text
ns-100.awsdns.com
ns-200.awsdns.net
ns-300.awsdns.org
ns-400.awsdns.co.uk
```

You copy them into GoDaddy.

GoDaddy updates the .com registry.

Now the .com database contains:

```text
xyz.com

NS:
ns-100.awsdns.com
ns-200.awsdns.net
ns-300.awsdns.org
ns-400.awsdns.co.uk
```

---

# Chapter 10: What Is a Resolver?

Now the most important component.

Suppose user types:

```text
api.xyz.com
```

Browser doesn't perform DNS lookup itself.

It asks a resolver.

Example:

* Google DNS (8.8.8.8)
* Cloudflare DNS (1.1.1.1)

Think:

```text
Resolver = Detective
```

---

# Chapter 11: Complete Lookup

User enters:

```text
api.xyz.com
```

---

### Step 1

Browser asks Resolver:

```text
What is api.xyz.com?
```

---

### Step 2

Resolver asks Root:

```text
Who handles .com?
```

Root replies:

```text
Ask .com servers
```

---

### Step 3

Resolver asks .com server:

```text
Who manages xyz.com?
```

.com database:

```text
xyz.com

NS:
ns-100.awsdns.com
ns-200.awsdns.net
```

Reply:

```text
Ask Route53
```

---

### Step 4

Resolver asks Route53:

```text
What is api.xyz.com?
```

Route53 opens Hosted Zone:

```text
api.xyz.com → 54.23.11.101
```

Returns:

```text
54.23.11.101
```

---

### Step 5

Resolver returns answer to browser.

Browser now knows:

```text
api.xyz.com → 54.23.11.101
```

---

### Step 6

Browser opens TCP connection:

```text
54.23.11.101:443
```

---

### Step 7

HTTPS handshake.

---

### Step 8

HTTP Request:

```http
GET /
Host: api.xyz.com
```

---

### Step 9

Server responds.

# Never Forget Table

| Component   | Job                                 |
| ----------- | ----------------------------------- |
| Browser     | Needs IP                            |
| Resolver    | Finds IP                            |
| Root Server | Knows TLD servers                   |
| .com Server | Knows domain nameservers            |
| Nameserver  | Knows DNS records                   |
| Hosted Zone | Stores DNS records                  |
| Route53     | Provides nameservers + hosted zones |
| GoDaddy     | Sells domains and updates registry  |

# The 5 Things You Must Memorize

```text
Registrar  = Sells domain

Resolver   = Finds answer

Root Server = Finds TLD

TLD Server = Finds Nameserver

Nameserver = Finds IP from Hosted Zone
```

If these five roles are clear, you'll be able to understand advanced topics next, such as:

* A, AAAA, CNAME, MX, TXT records
* Recursive vs Iterative DNS lookup

* Load balancing with Route53
* Geo-routing, weighted routing, and failover routing
* How HTTPS and TLS use DNS (including SNI).
