# 🧠 Step 1: What is Parent–Child?

👉 In OOP:

* **Parent (Super class)** = General rule / contract
* **Child (Sub class)** = Specific implementation of that rule

---

## 🔹 Example

```java
class Bird {        // Parent
    public void eat() {
        System.out.println("Bird is eating");
    }
}

class Sparrow extends Bird {   // Child
    public void chirp() {
        System.out.println("Sparrow is chirping");
    }
}
```

👉 Here:

* `Bird` = Parent (common behavior)
* `Sparrow` = Child (special behavior)

---

# 🧠 Step 2: Real Meaning (VERY IMPORTANT)

👉 Parent says:

> “Any object of my type can do THIS”

👉 Child must follow that promise.

Let’s understand **L — Liskov Substitution Principle (LSP)** in a way you won’t forget.

---

# 🔥 What is LSP (Simple Definition)

👉 **“A child class should be able to replace its parent class without breaking the program.”**

In other words:

> If `B` is a subtype of `A`, then we should be able to use `B` **instead of** `A` without changing correctness.

---

# 🧠 Real-Life Intuition

Think:

* Parent: **Bird**
* Child: **Sparrow**, **Penguin**

If I say:
👉 “All birds can fly”

Then:

* Sparrow ✅ (flies)
* Penguin ❌ (does NOT fly)

➡️ So Penguin **breaks expectation**

👉 That means **LSP is violated**

---

# ❌ BAD EXAMPLE (Violation of LSP)

```java
class Bird {
    public void fly() {
        System.out.println("Flying...");
    }
}

class Sparrow extends Bird {
    // works fine
}

class Penguin extends Bird {
    @Override
    public void fly() {
        throw new UnsupportedOperationException("Penguin can't fly");
    }
}
```

---

## 🚨 Problem

```java
Bird bird = new Penguin();
bird.fly(); // 💥 Runtime error
```

👉 Why?

Because:

* Parent says: “I can fly”
* Penguin says: “No, I can’t”

➡️ Child **changed behavior**

👉 This breaks LSP ❌

---

# ✅ GOOD EXAMPLE (Following LSP)

👉 Fix: Separate behaviors properly

```java
class Bird {
    public void eat() {
        System.out.println("Eating...");
    }
}

class FlyingBird extends Bird {
    public void fly() {
        System.out.println("Flying...");
    }
}

class Sparrow extends FlyingBird {
    // can fly
}

class Penguin extends Bird {
    // no fly method
}
```

---

## ✅ Now Usage is Safe

```java
Bird bird1 = new Sparrow();
bird1.eat(); // ✅ OK

Bird bird2 = new Penguin();
bird2.eat(); // ✅ OK
```

```java
FlyingBird bird3 = new Sparrow();
bird3.fly(); // ✅ OK
```

👉 No unexpected behavior
👉 No runtime crash
👉 LSP is followed ✅

---

# 🧠 Key Rule (VERY IMPORTANT)

👉 **Child should NOT:**

* Throw new exceptions
* Remove expected behavior
* Change meaning of parent method

# 💡 One More Real Example (Very Important for Interviews)

## ❌ BAD (Bank Account Example)

```java
class BankAccount {
    public void withdraw(int amount) {
        System.out.println("Withdrawn: " + amount);
    }
}

class FixedDepositAccount extends BankAccount {
    @Override
    public void withdraw(int amount) {
        throw new UnsupportedOperationException("Cannot withdraw before maturity");
    }
}
```

---

## 🚨 Problem

```java
BankAccount account = new FixedDepositAccount();
account.withdraw(1000); // 💥 Crash
```

👉 Parent allows withdraw
👉 Child denies it ❌

---

# ✅ GOOD DESIGN

```java
class BankAccount {
    public void deposit(int amount) {
        System.out.println("Deposited: " + amount);
    }
}

interface Withdrawable {
    void withdraw(int amount);
}

class SavingsAccount extends BankAccount implements Withdrawable {
    public void withdraw(int amount) {
        System.out.println("Withdrawn: " + amount);
    }
}

class FixedDepositAccount extends BankAccount {
    // no withdraw
}
```
